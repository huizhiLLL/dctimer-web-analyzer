# DCTimer Web Analyzer UI 设计基线

## 1. 设计目标

网站前端第一版界面需要满足：
- 移动端友好
- 简约
- 高级感
- 数据产品气质明显
- 适合后续承载“年度总结 / 数据报告 / 成就卡片”

虽然使用了 `ui-ux-pro-max` 的设计建议，但不直接照搬“重 dashboard”的感觉，而是做一次产品化收束。

---

## 2. 风格方向

最终采用：

**Minimal Premium Data Report**

关键词：
- 极简
- 克制
- 深色 / 浅色都可扩展
- 卡片化
- 数据感
- 高级留白
- 微发光强调
- 移动端优先

避免：
- 过度游戏 UI
- 太炫的霓虹风
- 太硬核后台管理味
- 大量装饰性渐变和玻璃拟态

目标不是“管理后台”，而是“高级数据作品”。

---

## 3. 视觉基调

### 3.1 色彩

基础方向：
- 主色：深蓝 / 靛蓝
- 强调色：偏琥珀金 / 暖黄
- 中性色：大面积冷灰白 / 深灰蓝

建议色板：
- Primary: `#1E40AF`
- Primary Soft: `#3B82F6`
- Accent: `#F59E0B`
- Background: `#F8FAFC`
- Surface: `#FFFFFF`
- Text Strong: `#0F172A`
- Text Muted: `#475569`
- Border: `#E2E8F0`

后续深色模式可扩展为：
- Background: `#0B1220`
- Surface: `#111827`
- Text Strong: `#E5E7EB`
- Text Muted: `#94A3B8`

### 3.2 气质

整体更像：
- Apple 年报 + 数据产品
- Linear 式克制
- 少量游戏化成就语言

而不是：
- 传统 BI 后台
- 电竞风 UI
- 彩色拼贴风

---

## 4. 布局原则

### 4.1 移动端优先

优先以手机竖屏设计：
- 375px 起步
- 单列布局优先
- 桌面端再扩展为多列

### 4.2 页面宽度

建议：
- 内容容器：`max-width: 1200px`
- 报告类正文区：`max-width: 960px`
- 超宽图表再允许单独突破

### 4.3 间距系统

建议采用 4/8 基准：
- 4 / 8 / 12 / 16 / 24 / 32 / 48 / 64

移动端重点：
- 页面左右 padding 不少于 16px
- 卡片内边距不小于 16px
- 主要点击区域高度不少于 44px

---

## 5. 组件风格原则

### 5.1 卡片

卡片应该：
- 圆角中等偏大
- 轻边框
- 轻阴影
- 少量 hover 强调
- 信息层次清晰

建议：
- `border-radius: 20px`
- `border: 1px solid #E2E8F0`
- `box-shadow: 0 8px 30px rgba(15, 23, 42, 0.06)`

### 5.2 按钮

主按钮：
- 实色主色背景
- 清晰 hover / active / disabled 状态

次按钮：
- 白底描边
- 更适合筛选、切换

要求：
- 不用 emoji 图标
- 图标统一使用 SVG / Lucide
- 最小点击面积 44x44

### 5.3 图表容器

图表不直接裸放。建议：
- 每张图都有标题
- 每张图都有简短解释文字
- 图表包在统一卡片里
- 移动端优先纵向堆叠

---

## 6. 字体建议

`ui-ux-pro-max` 给出的 Fira Code + Fira Sans 更偏技术仪表盘。

小枝这里建议做一点收束：

### 推荐方案
- 标题：`Inter` / `Manrope`
- 正文：`Inter`
- 数字：可局部用 `JetBrains Mono` 或 `Fira Code`

这样更符合：
- 简约
- 高级感
- 数据可读性
- 移动端显示稳定

### 使用原则
- 只有数字 KPI / code / 时间串 用等宽字体
- 标题和正文不要全站等宽，否则太工程味

---

## 7. 动效原则

动效要有，但要轻。

建议：
- 页面切换：淡入 + 上移 8px
- 卡片 hover：阴影加强 / 边框强调
- 数字加载：滚动或渐显
- 图表出现：opacity + translateY

避免：
- 大面积弹跳
- 高频闪烁
- 强缩放 hover

时长建议：
- 微交互：150~220ms
- 页面模块进入：220~320ms

并且要支持：
- `prefers-reduced-motion`

---

## 8. 页面级建议

### 8.1 首页 / 导入页

感觉应该像一个精致的上传入口，而不是文件管理器。

建议模块：
- 标题区
- 副标题说明“纯前端、本地解析、不上传”
- 上传卡片
- 功能亮点 3 卡片

### 8.2 分组筛选页

要解决“多分组选择”的复杂性。

建议：
- 顶部固定筛选区
- 下方 session 卡片列表
- 每张卡片展示：
  - 分组名
  - 项目类型
  - 解次
  - 时间范围
- 支持 checkbox 或整卡选中

### 8.3 报告页

报告页更像“故事流”。

顺序建议：
1. Hero 总结
2. KPI 卡片
3. 时间趋势
4. 分组画像
5. 高光时刻
6. 成就 / 分享卡

移动端必须保证：
- 一屏一重点
- 不要塞太多图
- 文案和数字比例平衡

---

## 9. 可访问性与交互底线

必须遵守：
- 文本对比度达到 4.5:1
- 可点击区域 >= 44x44
- 焦点态清晰
- 图标按钮带 aria-label
- 图表尽量有文字总结辅助
- 移动端不出现横向滚动

---

## 10. 前端实现建议

当前工程建议：
- Vue 3 + TS + Vite
- 路由：Vue Router
- 状态：Pinia
- 样式：建议 Tailwind CSS
- 图标：Lucide
- 图表：ECharts

原因：
- Tailwind 很适合快速建立统一设计 token
- 移动端响应式处理快
- 做卡片、布局、状态样式很稳

---

## 11. 首版组件清单

建议优先做：
- AppShell
- PageHeader
- UploadCard
- FilterToolbar
- SessionSelectCard
- KpiCard
- ChartCard
- StoryCard
- EmptyState
- LoadingState

---

## 12. 结论

首版 UI 方向应该定为：

**移动端优先的简约高级数据报告网站**

表达方式上：
- 用克制的视觉做高级感
- 用少量成就化语言提供作品感
- 用清晰的卡片和图表承载复杂数据

后面如果要再增加“游戏化”，应该更多放在：
- 成就命名
- 高光卡片
- 动效节奏

而不是把整站做成游戏面板。
